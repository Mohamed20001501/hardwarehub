01 · Setup

Get it running locally

Five steps, run in order. The backend and frontend are separate npm projects (server/ and client/) that talk over HTTP; MongoDB must be reachable before the backend starts.

  1. Start MongoDB

    The app expects a database at mongodb://127.0.0.1:27017/ecommerce (configurable in server/.env).

    # if installed via Homebrew
brew services start mongodb-community
# or run it directly
mongod --dbpath /path/to/data

  2. Install and configure the backend

    server/.env ships with working local defaults, so this step usually needs no edits.

    cd server
npm install

  3. Seed the database

    Admin isn't self-registerable, so it's created by script. Sample vendors/customers/products/orders and product images are optional but recommended for a populated first run.

    npm run seed          # creates the admin account
npm run seed:data     # sample vendors, customers, products, orders
npm run seed:images   # generates product icon images

  4. Start the API

    npm run dev   # http://localhost:5001

  5. Install and start the frontend

    In a second terminal:

    cd client
npm install
npm run dev   # http://localhost:5173

    Open http://localhost:5173 — that's the whole app.

03 · User guide

Three roles, one storefront

Every account has exactly one role, fixed at registration. Customers and vendors self-register and choose their role on the sign-up form; the admin role is never exposed there — it exists only for the account created by npm run seed.

Customer

Browses & buys

  • Search and sort the shared catalog
  • Cart persists locally between visits
  • Places orders, tracks status
Vendor

Lists & fulfills

  • Owns a private product catalog
  • Uploads a photo per product
  • Advances status on its own orders only
Admin

Oversees everything

  • Platform-wide stats and revenue
  • Manages every user, product, order
  • Cannot delete the admin account itself
Customer

Browse to delivered

The path a shopper takes from landing on the homepage to holding a confirmed order.

Create an account

On /register, fill name, email, password, and select Customer. Submitting logs you in immediately — no separate confirmation step.

Find a product

On /products, type into the search box (matches name, description, or category) or reorder by price with the sort dropdown. Out-of-stock items are labeled and their Add to Cart is disabled.

Add to cart

From a product page, choose a quantity (capped at the vendor's current stock) and add it. The cart lives in localStorage — it survives a refresh but not a different browser or device.

Review the cart

On /cart, adjust quantities or remove lines. The subtotal recalculates live.

Check out

On /checkout, confirm the order total and place it. There's no payment form here — the proposal explicitly scopes online payment out of this version, so placing the order is the final step.

Track it

On /orders, each past order shows its line items, total, and current status.

Pending→ Processing→ Delivered
Vendor

List a product, fulfill an order

Vendors only ever see and act on their own inventory and the orders that contain it.

Register as a vendor

Same form as a customer, with Vendor selected. This redirects straight to the Vendor Dashboard instead of the storefront home.

Check the overview

/vendor shows three live counters: products listed, orders received, and estimated revenue from those orders.

Add a product

Under My Products, + Add Product opens a form for name, category, price, stock, description, and an image file. The image is optional but strongly recommended — products without one show a plain "No image" placeholder to shoppers.

Edit or retire a listing

Each row has Edit (same form, pre-filled — leave the image field empty to keep the existing photo) and Delete (immediate, with a confirmation prompt).

Fulfill an order

Under Orders, every order containing at least one of your products appears with the buyer's name and the full line-item list. Move the status dropdown from Pending to Processing to Delivered as the order actually progresses — the customer sees the update immediately on their next visit to Order History.

Admin

Oversee the marketplace

One account, created only via npm run seed, with visibility across every vendor and customer.

Sign in

Use the credential from server/.env (defaults printed in Sample data below) at /login. There's no admin sign-up path by design.

Read the overview

/admin totals users (split by role), products, orders, and cumulative revenue across the whole platform.

Manage users

Users lists every account with its role. Any customer or vendor can be deleted; the admin row has no delete control — the backend refuses that call even if attempted directly.

Manage products

Products lists every vendor's catalog in one table, with the owning vendor's name attached, and a direct delete action that bypasses ownership checks.

Manage orders

Orders lists every order platform-wide and lets the admin set status directly — useful for resolving a case a vendor hasn't acted on.

06 · Troubleshooting

When a step doesn't do what this doc says

Backend crashes with EADDRINUSE on port 5000

On macOS, the AirPlay Receiver (Control Center) listens on port 5000 by default. This project's backend already defaults to 5001 for exactly this reason — if you changed PORT back to 5000, either pick a different port or turn off AirPlay Receiver in System Settings → General → AirDrop & Handoff.

Frontend loads but every request fails / CORS error in the console

Check two things line up: CLIENT_URL in server/.env must exactly match the URL the frontend is actually served from (protocol, host, and port), and the frontend's API base URL (client/src/api/axios.js) must point at the backend's actual port.

Login succeeds but refreshing the page logs you out

Auth is a JWT stored in an httpOnly cookie, restored on load via GET /api/auth/me. If that call fails silently, check the backend is actually running and that the browser isn't blocking third-party cookies between the two localhost ports (Safari's default tracking prevention can do this — Chrome and Firefox do not, for localhost).

Product images don't show, just a "No image" placeholder

Either the product was created without a file attached, or npm run seed:images hasn't been run yet for sample data. Uploaded files live in server/uploads/ — confirm the folder isn't empty and that the backend's static file route is reachable at http://localhost:5001/uploads/<filename>.

mongod won't start, or the app can't connect to MongoDB

Confirm nothing else is already bound to port 27017 (lsof -i :27017), and that the data directory passed to --dbpath exists and is writable. If MongoDB is already running as a background service, you don't need to start it again — just confirm it's up before running npm run seed.

Placing an order fails with "Insufficient stock"

Stock is decremented at order time and re-validated against the database, not just the number shown in the UI. If two customers try to buy the last unit of the same item, the second checkout will correctly fail this way rather than overselling.
